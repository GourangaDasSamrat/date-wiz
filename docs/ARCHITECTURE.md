# Architecture

This document explains how `date-wiz` is structured internally, the decisions behind its design, and how the pieces fit together.

---

## Goals

- **Zero dependencies** — only native `Date` and `Intl` objects
- **Immutability** — every operation returns a new value; inputs are never mutated
- **Tree-shakability** — ES Module design lets bundlers drop unused code
- **TypeScript-first** — full type safety, no `any`, no casting hacks
- **Graceful degradation** — invalid inputs return a fallback, never throw or crash

---

## Repository layout

```
date-wiz/
├── src/                  # TypeScript source (the only thing you should edit)
│   ├── types.ts          # All public-facing types and interfaces
│   ├── utils.ts          # Internal helpers (never exported publicly)
│   ├── format.ts         # format()
│   ├── relative.ts       # getRelativeTime()
│   ├── smart.ts          # smartFormat()
│   ├── parse.ts          # parse()
│   ├── arithmetic.ts     # add / subtract / diff / startOf / endOf / …
│   ├── business.ts       # addBusinessDays / isWithinWorkingHours / …
│   ├── wiz.ts            # wiz() chainable factory
│   └── index.ts          # Public barrel — re-exports everything
│
├── dist/                 # Generated by `npm run build` — do not edit
│   ├── esm/              # ES Modules (tree-shakable, for bundlers)
│   ├── cjs/              # CommonJS (for Node `require()`)
│   └── types/            # TypeScript declaration files (.d.ts)
│
├── tests/                # Jest test suites, one file per source module
├── scripts/
│   └── post-build.mjs    # Injects package.json markers into dist/esm & dist/cjs
├── docs/                 # Project documentation (not shipped in the npm package)
└── .github/workflows/    # CI and publish pipelines
```

## Module dependency graph

```
index.ts  (barrel)
  ├── wiz.ts
  │     ├── format.ts
  │     ├── smart.ts
  │     ├── relative.ts
  │     └── arithmetic.ts
  │           └── business.ts
  ├── format.ts
  ├── relative.ts
  ├── smart.ts
  │     └── format.ts
  ├── parse.ts
  ├── arithmetic.ts
  │     └── business.ts
  └── business.ts
        └── utils.ts

utils.ts  ← depended on by almost everything; never imports from siblings
types.ts  ← pure type declarations; no runtime code
```

No module outside `utils.ts` and `types.ts` is imported by more than two siblings, keeping coupling low and making each module independently usable.

## Key design decisions

### No `Date` mutation

Every function that produces a new date calls `clone()` from `utils.ts` before modifying anything. `clone()` is the single chokepoint for immutability — if you add a new operation, always start with `const result = clone(d)`.

### `Intl` over hand-rolled locale data

Month names, weekday names, and relative time phrases are all produced by the built-in `Intl.DateTimeFormat` and `Intl.RelativeTimeFormat` APIs. This eliminates locale files entirely and keeps the bundle under 2 KB gzipped while supporting 100+ languages.

### Token-based formatting without a parser

`format.ts` uses a single regex (`TOKEN_RE`) that matches all supported tokens in one pass. Each matched token is replaced via a pre-computed lookup map. The regex is defined once at module scope so it is compiled only once per process.

### Sub-path exports for tree-shaking

`package.json` declares an `exports` map with sub-paths (`date-wiz/format`, `date-wiz/relative`, etc.). This means a consumer who imports only `format` will have the relative-time, parse, and business-day code completely eliminated by their bundler's tree-shaking.

### Three output formats

| Format | Location      | Used by                          |
| ------ | ------------- | -------------------------------- |
| ESM    | `dist/esm/`   | Bundlers (Vite, webpack, Rollup) |
| CJS    | `dist/cjs/`   | Node.js `require()`              |
| Types  | `dist/types/` | TypeScript compiler              |

Each `dist/esm/` and `dist/cjs/` directory contains a `package.json` with `"type": "module"` or `"type": "commonjs"` respectively, so Node resolves them correctly without needing `"type": "module"` in the root `package.json`.

## Build pipeline

```
npm run build
  │
  ├─ tsc -p tsconfig.esm.json   →  dist/esm/*.js
  ├─ tsc -p tsconfig.cjs.json   →  dist/cjs/*.js
  ├─ tsc -p tsconfig.types.json →  dist/types/*.d.ts
  └─ node scripts/post-build.mjs
         writes dist/esm/package.json  { "type": "module" }
         writes dist/cjs/package.json  { "type": "commonjs" }
```

The `prepublishOnly` hook in `package.json` runs `npm run build` automatically before every `npm publish`, so the `dist/` folder in the published package is always fresh.

## Adding a new utility

1. Create `src/my-util.ts`. Import only from `./utils.js` or `./types.js` unless there is a clear dependency need.
2. Export it from `src/index.ts`.
3. Add a sub-path entry in the `exports` field of `package.json` if it deserves standalone tree-shakable access.
4. Write `tests/my-util.test.ts`. Aim for 100% function coverage.
5. Document the public API in `README.md`.
